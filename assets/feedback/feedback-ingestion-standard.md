# Feedback Ingestion Standard

## Purpose

Define how consumer repos capture and process feedback markdown so guidance can
improve continuously without losing human context.

## Source Of Truth

Markdown files inside the consumer repo are the source of truth for feedback.

Preferred locations:

- `.ai-native/feedback/toil/`
- `.ai-native/feedback/developer-notes/`
- `.ai-native/audits/`

Equivalent repo-local locations are acceptable if the repo has a stronger
existing convention.

## What Counts As Feedback

The following should be treated as ingestible feedback:

- markdown generated by an agent when toil, confusion, repeated retries, or
  avoidable rework happened during delivery
- markdown created by developers in isolation that describes friction, desired
  guidance, missed expectations, or quality concerns
- audit markdown created during onboarding, refactors, UI reviews, or standards
  checks

## Automatic Capture Rule

If a task encounters toil that better guidance could have prevented, the agent
should leave behind a feedback markdown file before considering the task done.

Examples:

- ambiguous workflow expectations
- missing UI review criteria
- recurring spacing or layout corrections
- repeated setup or environment mistakes
- confusion around modularity or contract boundaries

## Ingestion Cadence

Use two default ingestion points:

1. Commit-time sweep
   Review new or changed markdown under feedback and audit directories before a
   branch is pushed or merged.
2. Scheduled cadence
   Run a recurring review, typically weekly, to cluster recent feedback into
   shared themes and decide whether local rules or shared AI Native assets
   should change.

Consumer repos may add stronger cadences, but they should not remove both of
these default checkpoints.

## Promotion Path

1. Capture the feedback in repo-local markdown.
2. Triage whether it is repo-specific or likely cross-repo.
3. Convert recurring local issues into repo-local standards, checklists, or
   workflows.
4. Promote cross-repo patterns into AI Native assets after human review.

## Human Gates

Human review is required for:

- promoting repo-local feedback into shared AI Native assets
- changing approval rules, workflow requirements, or protected-branch behavior
- tightening standards that could materially slow delivery

Routine capture, clustering, and draft recommendations may continue
automatically.

## Done Criteria

- feedback is stored in markdown inside the repo
- commit-time and scheduled ingestion expectations are explicit
- toil-triggered feedback is expected, not optional
- isolated developer markdown is treated as signal
